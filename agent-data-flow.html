🔄 Agent 数据流转图

从用户输入到 AI 响应的完整数据路径 — State 怎么变、消息怎么流

1. 总览：谁读写 State

┌─────────────────────────────────────────────────────────────────┐
│                       ThreadState（总账本）                      │
│                                                                 │
│  messages:    [human, ai, tool, ai, ...]    ← 中间件读写        │
│  sandbox:     { sandbox_id: "..." }         ← sandbox_middleware │
│  thread_data: { workspace, uploads, outputs } ← thread_data    │
│  delegations: [{ id, status, result }]      ← durable_context   │
│  skill_context: [{ name, path }]            ← durable_context   │
│  summary_text: "用户要求分析..."            ← summarization     │
│  viewed_images: { "path": { meta } }        ← view_image_tool   │
│  artifacts: ["/mnt/.../report.md"]          ← present_file_tool  │
│  uploaded_files: [{ filename, size }]       ← uploads middleware │
│  title: "销售数据分析"                      ← title middleware   │
│  todos: [{ content, status }]               ← todo middleware    │
└─────────────────────────────────────────────────────────────────┘

2. 一轮完整对话的数据流

每一步数据长什么样？state 里的 messages 怎么变的？ 

// ===== 第 0 步：初始 State =====
{
  "messages": [],
  "sandbox": null,
  "thread_data": null,
  "delegations": [],
  "skill_context": [],
  "summary_text": null
}
⬇︎ 用户发消息
// ===== 第 1 步：用户输入 → 中间件预处理 =====

// thread_data 中间件写入：
thread_data: {
  workspace_path: ".deer-flow/threads/abc/user-data/workspace",
  uploads_path:   ".deer-flow/threads/abc/user-data/uploads",
  outputs_path:   ".deer-flow/threads/abc/user-data/outputs"
}

// uploads 中间件检测到上传文件，写入：
uploaded_files: [{ filename: "data.xlsx", size: 10240, path: "/mnt/.../data.xlsx" }]

// input_sanitization 检查消息内容：
// "帮我分析 data.xlsx" → 没有 <system> 标签 → 原样通过

// sandbox_middleware 初始化沙箱：
sandbox: { sandbox_id: "local:default:abc" }
⬇︎ 第 1 次调 LLM
// ===== 第 2 步：AI 回复 + 工具调用 =====

// 现在 messages 里有：
[
  { type: "human", content: "帮我分析 data.xlsx" }, ← 用户输入
  { type: "ai", content: null, tool_calls: [← AI 说要调工具
    { id: "call_001", name: "read_file", args: { path: "/mnt/.../data.xlsx" } }
  ]},
  { type: "tool", content: "日期,销售额,地区\n2025-01,100\n...", ← 工具返回结果
    tool_call_id: "call_001" }
]
⬇︎ AI 继续推理
// ===== 第 3 步：AI 回复完成 =====

[
  { type: "human", content: "帮我分析 data.xlsx" },
  { type: "ai", tool_calls: [{ id: "call_001", name: "read_file" }] },
  { type: "tool", content: "日期,销售额,地区...", tool_call_id: "call_001" },
  { type: "ai", content: "数据有三列：日期、销售额、地区" } ← AI 最终回复
]

// 后处理中间件：
// title → 检测到第一次对话 → 生成标题 "销售数据分析"
title: "销售数据分析"

// durable_context → 没有子代理，没有技能读取 → 无操作

// memory_middleware → 把对话加入记忆更新队列
// 30 秒后 LLM 提取事实："用户分析了销售数据"

3. 带子代理的复杂数据流

// ===== 第 0 步：用户指派子代理 =====

messages: [
  { type: "human", content: "分别搜索三个城市的销售数据" },
  { type: "ai", tool_calls: [
    { id: "c1", name: "task", args: { description: "搜索北京数据" } },
    { id: "c2", name: "task", args: { description: "搜索上海数据" } },
    { id: "c3", name: "task", args: { description: "搜索广州数据" } }
  ]}
]

// 子代理执行过程中，durable_context 逐步捕获：

delegations: [
  { id: "task_001", description: "搜索北京数据", status: "running" },
  { id: "task_002", description: "搜索上海数据", status: "running" },
  { id: "task_003", description: "搜索广州数据", status: "running" }
]
⬇︎ 子代理陆续返回
// ===== 第 1 步：子代理结果捕获 =====

// 第一个子代理完成 → durable_context 检测到变化
delegations: [
  { id: "task_001", description: "搜索北京数据",
    status: "completed", result_brief: "销售额100万" },
  { id: "task_002", description: "搜索上海数据", status: "running" },
  { id: "task_003", description: "搜索广州数据", status: "running" }
]

// 第一个子代理完成 → durable_context 检测变化，只追加变化的部分
// _filterChangedDelegations → task_001 状态从 running → completed
// 只更新 task_001，不碰 task_002 和 task_003
⬇︎ 全部完成，AI 汇总
// ===== 第 2 步：AI 汇总结果 =====

delegations: [
  { id: "task_001", description: "搜索北京数据", status: "completed", result_brief: "销售额100万" },
  { id: "task_002", description: "搜索上海数据", status: "completed", result_brief: "销售额200万" },
  { id: "task_003", description: "搜索广州数据", status: "completed", result_brief: "销售额300万" }
]
⬇︎ 对话太长，被压缩
// ===== 第 3 步：压缩后，injectDurableContext 救回数据 =====

// summarization 把中间消息压缩了，messages 变短
// 但 delegations 还在 state 里（state 不会被压缩）

// injectDurableContext 在每次 LLM 调用前插入：
{ type: "human", content: "<durable_context_data>
## Conversation summary so far
用户要求搜索三个城市的销售数据...

## Delegation ledger
- task_001: 搜索北京数据 → completed
- task_002: 搜索上海数据 → completed
- task_003: 搜索广州数据 → completed
</durable_context_data>", hide_from_ui: true }

4. 中间件按顺序读写 State

输入 → input_sanitization     → 读 messages（检查注入），不改 state
       thread_data             → 写 thread_data（计算路径，可选创建目录）
       uploads                  → 写 uploaded_files（注入上传文件信息）
       sandbox_middleware       → 写 sandbox（获取沙箱）
       dynamic_context          → 插 messages（注入日期 + 记忆）
       skill_activation         → 插 messages（注入技能内容）
       ──────── 调 LLM ────────
       tool_progress            → 读 messages + tool_meta（检测空转）
       tool_error_handling      → 写 tool_meta（标记错误类型）
       read_before_write        → 读 messages（检查写前读）
       durable_context          → 写 delegations + skill_context
       token_usage              → 写 messages[].additional_kwargs（归因标签）
       title                     → 写 title（第一次对话时生成）
       memory_middleware         → 读 messages，加入记忆队列（30 秒后更新）
       summarization             → 写 summary_text + 删 messages（太长时压缩）
输出 ←

5. 关键：State 和 Messages 的关系

// 💡 记住这三点：

1. messages 在 state 里面
   state = { messages: [...], sandbox: {...}, thread_data: {...}, ... }
             ↑ messages 是 state 的一个字段

2. 中间件可以读 messages（检查内容）也可以写 messages（插入新消息）
   - input_sanitization → 读 messages，不改（只是检查）
   - dynamic_context → 写 messages（插入日期提醒）
   - durable_context → 写 messages（注入持久上下文）

3. 中间件也可以读写 state 的其他字段
   - thread_data → 写 state.thread_data（不碰 messages）
   - durable_context → 写 state.delegations（不碰 messages）
   - summarization → 写 state.summary_text + 删 messages（改 messages 也改其他字段）

6. 完整的 messages 数据结构

// 消息类型有三种，按顺序排列：

HumanMessage（用户）：
{ type: "human", id: "msg_001", content: "帮我分析数据", name: "user-input",
  additional_kwargs: { run_id: "run_123", timestamp: "2026-07-23T10:00:00Z" } }

AIMessage（AI 回复）：
{ type: "ai", id: "msg_002", content: "好的我要先读文件",
  tool_calls: [{ id: "call_001", name: "read_file", args: { path: "..." } }],
  usage_metadata: { input_tokens: 150, output_tokens: 30, total_tokens: 180 },
  additional_kwargs: { token_usage_attribution: { version: 1, kind: "tool_batch",
    actions: [{ kind: "tool", tool_name: "read_file" }] } } }

ToolMessage（工具结果）：
{ type: "tool", id: "msg_003", content: "日期,销售额...",
  tool_call_id: "call_001", name: "read_file", status: "success" }

// 隐藏消息（hide_from_ui=true，用户看不到）：
{ type: "human", content: "<durable_context_data>...</durable_context_data>",
  additional_kwargs: { hide_from_ui: true, durable_context_data: true } }

{ type: "human", content: "<slash_skill_activation>...</slash_skill_activation>",
  additional_kwargs: { hide_from_ui: true, slash_skill_activation: true } }
